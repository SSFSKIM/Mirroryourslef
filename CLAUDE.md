## Project Overview

**MirrorYourself** (also known as ViewTime / Y-Stats) is a privacy-first YouTube analytics application that helps users understand their viewing habits. The UI is branded MirrorYourself; some docs and filenames still say ViewTime.

The app lives under `view-time (5)/`. Everything outside that directory is configuration or documentation.

## Architecture

**Full-stack application:**
- **Frontend**: React 18 + TypeScript + Vite, shadcn/ui components, Tailwind CSS
- **Backend**: FastAPI (Python 3.13+ via `uv`, runtime uses Python 3.11-slim in Docker)
- **Storage**: KV abstraction (`app/libs/kv_store.py`) ‚Äî prefers Firestore, falls back to Databutton storage
- **Authentication**: Firebase Auth (Google OAuth) + Firebase Admin SDK for backend token validation
- **Deployment**: Databutton platform (production), Docker (local), or Google Cloud Run

## Development Commands

> **Note:** There is no `Makefile` in this repo. Ignore references to `make` in older docs.

### Local Development (recommended)

```bash
# Backend
cd "view-time (5)/backend"
./install.sh        # Creates venv, installs dependencies
./run.sh            # Activates venv, runs uvicorn main:app --reload on port 8000

# Frontend (in a separate terminal)
cd "view-time (5)/frontend"
./install.sh        # corepack enable + yarn install
./run.sh            # yarn dev on port 5173
```

### Docker: Development with Hot-Reload

```bash
cd "view-time (5)"
docker compose -f docker-compose.dev.yml up --build
# Backend on port 8000, Frontend on port 5173
```

### Docker: Production-Like

```bash
cd "view-time (5)"
docker compose up --build
# Combined Nginx container on port 80
```

### Frontend Only

```bash
cd "view-time (5)/frontend"
yarn dev        # Start development server
yarn build      # Build for production
yarn lint       # Run ESLint
```

## Key Technologies & Patterns

### Frontend
- **State Management**: Zustand stores:
  - `useAuthStore` (`utils/auth.ts`) ‚Äî Firebase sign-in state
  - `dataStore` (`utils/dataStore.ts`) ‚Äî liked-videos sync status, analytics, user profile
  - `watchHistoryStore` (`utils/watchHistoryStore.ts`) ‚Äî watch history upload/analytics
- **UI Components**: shadcn/ui primitives under `extensions/shadcn/`
- **Styling**: Tailwind CSS with dark mode (default dark via `ThemeProvider`)
- **Charts**: Recharts for analytics visualizations
- **Authentication**: Firebase Auth ‚Äî reuse `firebaseApp`, `firebaseAuth`, `firestore`, `UserGuard`, `useCurrentUser` from `app` exports. **Do not initialize a second Firebase app.**
- **API Client**: Generated Brain client in `src/brain/` ‚Äî do not modify manually. Active methods:
  - `brain.sync_liked_videos(...)`, `brain.get_sync_status()`, `brain.get_analytics(...)`, `brain.get_user_summary()`
  - `brain.get_watch_history_status()`, `brain.get_watch_history_analytics()`, `brain.upload_watch_history_takeout(...)`, `brain.delete_watch_history()`
- **Direct fetch**: Only acceptable for third-party APIs (e.g., YouTube validation in `YouTubeSyncButton.tsx`)

### Backend
- **App Bootstrap**: `main.py` dynamically imports routers from `app/apis/*/__init__.py`
- **Router Auth**: Controlled by `routers.json` ‚Äî new modules must be added there
- **Authentication**: Firebase JWT validation via `databutton_app/mw/auth_mw.py`. All current routers require auth. Use `AuthorizedUser` from `app.auth` on authenticated endpoints.
- **Storage**: `app/libs/kv_store.py` KV abstraction ‚Äî prefers Firestore, falls back to Databutton storage. Not purely Firestore-collection-driven.
- **Logging**: Uses `print(...)`, not Python `logging` module
- **Models**: Pydantic models for request/response validation

### Adding a New API Module
1. Create `app/apis/<name>/__init__.py` with `router = APIRouter(...)`
2. Add entry to `backend/routers.json`
3. If missing from `routers.json`, bootstrap/auth wiring will not work

## Core Features (Implemented)

### Liked Videos Analytics
- Frontend collects Google OAuth access token with `youtube.readonly` scope
- Backend fetches likes playlist + per-video metadata from YouTube Data API
- Supported sample sizes: 50, 100, 150, 200, 250
- Dashboard shows: top keywords, category distribution, channel stats/loyalty, shorts vs regular, monthly trends, liked-time heatmap

### Watch History Analytics
- Users upload a Google Takeout `.json` or `.zip` file
- Backend parses watch events, sessionizes activity, computes summary metrics
- Dashboard shows: total events, unique videos/channels, session duration, recommendation breakdown, repeat views, heatmap, shorts share, algorithmic vs intentional split

### Smart Nudges
- **Rule-based frontend heuristics** from watch-history analytics
- **No active OpenAI integration** in the current code path

### Not Yet Implemented
- OpenAI-backed coaching, goal CRUD, achievements/streaks, community/social features, backend notification system. Some older code/docs mention these ‚Äî treat as planned/legacy.

## Important Conventions

### Frontend Cautions
The frontend is a hybrid of newer backend-driven analytics and older Firestore-first helpers. Files needing scrutiny before reuse:
- `frontend/src/utils/watchHistory.ts`
- `frontend/src/utils/database.ts`
- `frontend/src/utils/firestore.ts`
- Parts of `frontend/src/utils/dataStore.ts`

Prefer extending the current dashboard flow that uses backend routers and the Brain client.

### Security
- Never expose secrets in frontend code
- Never persist raw OAuth tokens beyond the minimum needed flow
- Keep YouTube API access in authenticated flows only
- Keep user-scoped data keyed by authenticated Firebase user ID
- Preserve the delete/reset behavior for watch-history data

## Testing & Quality

- **No automated test files** exist in the repo. Verification is lint + manual testing.

### Manual QA Checklist
1. Sign in with Google ‚Üí confirm `/dashboard` access
2. Run liked-videos sync ‚Üí confirm dashboard cards populate
3. Upload a Takeout `.json` or `.zip` ‚Üí confirm watch-history analytics appear

5. Check responsive layout and dark theme
6. If backend routes changed, verify Brain client still matches

## Deployment

### Local Development
1. Run `./install.sh` in both `backend/` and `frontend/`

3. Access at http://localhost:5173

### Docker Production
1. Copy `.env.docker.example` to `.env.docker` and fill in values
2. `docker compose up --build` from `view-time (5)/`
3. Access at http://localhost:80

- Use `deploy.sh` for deployment
- Environment config in `cloudrun.env.yaml`

### Databutton Platform
- Deploy from Databutton workspace UI
- Environment variables managed through platform secrets

## Known Drift & Pitfalls

- `docs/PRD_TRD.md` is partly aspirational ‚Äî not an exact description of what's built
- `README.md` references `make` but there is no `Makefile`
- `frontend/package.json` contains many template dependencies not central to the app
- `frontend/src/brain/*` includes stale legacy endpoints alongside active ones
- `frontend/vite.config.ts` injects Databutton defaults like `__APP_TITLE__ = "Databutton"`
- Some comments describe Databutton-only behavior even though the repo also supports Docker and Cloud Run
